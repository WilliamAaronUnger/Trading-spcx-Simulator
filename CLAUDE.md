# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

SPCX Trading-Duell: a two-player, pass-and-play stock trading game around a fictional SpaceX IPO. Both players trade the *exact same* pre-generated market one after the other; whoever ends with the higher P&L wins. It ships as an offline-capable PWA. All UI text, comments, and number formatting are in **German** (`de-DE` locale) ŌĆö keep new user-facing text and code comments in German to match.

## Development

There is no build system, package manager, linter, or test suite. The app is plain HTML/CSS/JS with zero dependencies.

- Run locally: serve the directory with any static server (e.g. `python3 -m http.server`) and open `index.html`. Opening the file directly also works, except the service worker only registers over http(s).
- There are no tests; verify changes by playing the game in a browser.

## Architecture

The application is split across four hand-maintained files (no build step): `index.html` (markup/structure only), `styles.css` (all CSS, formerly the inline `<style>`), `data.js` (the content **data** ŌĆö `STOCK_DEFS`/`ETF_DEF`, all news/event pools `NEWS_POOL`/`MEGA_POOL`/`MOMENTUM_POOL`/`CHAIN_POOL`/`GENERIC_*`/`OPENING_EVENT`, plus `TIPS`, `AWARDS`, `TUT_STEPS`, `MODE_HINTS`, `DURATIONS`, `DEFAULT_FAVS`, `KING_BADGE` and the tuning constants `TICK_MS`/`TICK_SCALE`/`REACT_TICKS`/`FEE_PCT`/`DIV_PCT`/`DIV_PAYOUT`/`MEGA_*` and the small data accessors `defOf`/`isDividendSym`/`DISPLAY_SYMS`/`feeOf`), and `game.js` (all logic: helpers, `mulberry32`/`genMarket`, state, tick loop, trading, rendering, news feed, round/match end, stats, tutorial, controls). The other files are PWA scaffolding (`sw.js`, `manifest.json`, icons).

These are **classic** scripts (not ES modules), loaded `data.js` **before** `game.js` at the end of `<body>`; both share one global lexical scope, exactly as the former single inline script did ŌĆö so top-level `const`/`let` in `data.js` are visible to `game.js` (and vice-versa at call time). No `import`/`export`. Keep `file://` opening working by not introducing modules. When adding/renaming any of these files, update `sw.js`'s `FILES` list **and** bump its `CACHE` version. `data.js` must stay free of parse-time references to `game.js` symbols (its internal order already satisfies dependencies, e.g. `TICK_MS` before `MEGA_REACT_TICKS`, `STOCK_DEFS` before `defOf`).

`game.js` is organized into commented sections (`/* ====== ... ====== */`) in this order: helpers, market generation, state, tick loop, trading, rendering, news feed, round/match end, controls.

### Core design: deterministic pre-generated market

The fairness guarantee of the game depends on one idea: the full market (all price paths and all news events) is generated **up front** in `genMarket(seed, ticks)` before round 1 starts, using a seeded PRNG (`mulberry32`). Both rounds then merely *replay* this same data via `tick()` (one tick every `TICK_MS` = 1000 ms; pool data is calibrated for 700 ms ticks and rescaled via `TICK_SCALE`). Consequences when editing:

- Anything random that affects prices or news must use the seeded `rnd()` inside `genMarket`, never `Math.random()`, or the two rounds diverge and the duel becomes unfair.
- `price(sym)` is just an index lookup into `market.paths[sym][tickCount]`; live game code never mutates prices.
- News events carry a `jump` (price shock) and a `drift` over `dur` ticks. The event is *displayed* at its `tick`, but the jump and drift only hit the price `REACT_TICKS` (~8 s) later ŌĆö that reaction window is the point; don't apply effects at the display tick.
- The 6-digit game code (`gameCode`) *is* the market seed: entering the same code on another device reproduces the identical game. `code % 3` encodes the duration (5/10/15 min via `DURATIONS`), so seed and tick count always match across devices. Don't change `genMarket`'s consumption order of `rnd()` casually ŌĆö it would silently change what every shared code produces.
- Creating a game opens a lobby overlay; round 1 starts automatically at the *second-next* full wall-clock minute (`openLobby`). Devices that create/join with the same code within the same minute therefore start simultaneously without any server.

### Other things to know

- There are three game modes, held in `mode`. The start screen presents a two-level chooser: a top row (`.mtop`, **Einzelspieler** / **Mehrspieler**) and, when Mehrspieler is picked, a sub row (`.mode`, **Gleiches Ger├żt** / **Zwei Ger├żte**); `setTop()` drives the top row and calls `setMode()`. A short `#modeHint` line (`MODE_HINTS`) explains the current pick. The three resulting `mode` values: **`"solo"`** (Einzelspieler) ŌĆö single player, one round, plays for themselves; **`"local"`** (Mehrspieler ┬Ę gleiches Ger├żt) ŌĆö the original pass-and-play: `players` has both entries, round 1 ŌåÆ handover overlay ŌåÆ round 2 ŌåÆ `showResult()` compares the two with a crown; **`"remote"`** (Mehrspieler ┬Ę zwei Ger├żte) ŌĆö one player per device coupled only by the shared code, lobby-timed start. Two axes cut across these: **timing** ŌĆö `"solo"` and `"local"` are tick-based and pausable (and their breaking-news popup pauses via `newsPaused`), only `"remote"` is wall-clock-anchored; **result** ŌĆö `"solo"` and `"remote"` run exactly one round (`round === 0`) and end via `showResultSolo()` (own P&L), but `"solo"` hides the result-exchange box (`cmpBox`) since there's no opponent. `"solo"` starts immediately (no lobby, random code) and counts toward the local record like `"local"`. Init calls `setTop("single")` so the default is Einzelspieler.
- Player state lives in the `players` array (cash, positions with average entry price, result). `round` indexes the active player.
- Positions support shorting: `pos.qty < 0` is a short with `avg` = average short price. There are three order buttons: buy (covers while short), sell (longs only), and a dedicated short button that opens/increases a short (capped by `maxShortQty`: open short exposure Ōēż 1├Ś `totalOf`, no leverage); an order never crosses from long to short or vice versa in one step. Generic formulas like `(cur - avg) * qty` already handle the sign ŌĆö but percent displays and chart up/down logic must flip for `qty < 0`. The result-exchange payload version (currently `SPCX3.ŌĆ”`) must be bumped whenever stats fields change.
- Game-time differs per mode: **local** is tick-based ŌĆö the pause button and the breaking-news popup (`newsPaused`) halt `tickCount`. **remote** is wall-clock-anchored ŌĆö `tick()` derives `tickCount` from `Date.now() - roundAnchor` (where `roundAnchor === startAt` from the lobby), there is no pause button, popups don't block (they auto-close), and skipped ticks are caught up after tab sleep. Anything that must stay in sync across devices has to key off this anchored `tickCount`.
- Each stock has a personality via extra `STOCK_DEFS` traits (`newsMult`, `momentum`, `meanRev`, `spikeP`/`spikeMag`), applied inside `genMarket`'s price loop ŌĆö still fully seeded. The `char` string is shown under the quote.
- Besides plain news there are two-stage chains (`CHAIN_POOL`: rumor ŌåÆ confirm/deny ~1.5ŌĆō2.5 min later) and insider tips (`market.tips`): vague heads-ups ~50 s before a real generated event. Tips derive from the seed, so both players/devices get identical ones ŌĆö fairness holds.
- Mega events (`MEGA_POOL`, ~50% of games, at most one, mid-game): huge jumps (+20ŌĆō40%, the negative Marktpanik capped at ~ŌłÆ15%) marked `mega:true` on the event. They are deliberately unannounced (excluded from insider-tip candidates) but use a longer fuse ŌĆö `MEGA_REACT_TICKS` (~20 s) instead of `REACT_TICKS` ŌĆö so shorts can still cover; that fuse is the safety valve, don't shorten it. `newsMult` is intentionally not applied to mega jumps. Jumps are not applied as a single tick: `pushJump()` ramps anything Ōēź4% over several ticks (`factor^(1/n)` per tick, ~7ŌĆō14 s for megas) so big moves rise as a curve ŌĆö the cumulative factor is unchanged, small news still hit in one tick, and no extra `rnd()` is consumed so fairness holds. Megas additionally get a **pre-tremor** (`fireMega`): `MEGA_PRE_FRAC` (~20%) of the move creeps in over the ~12 ticks *before* the main hit (same direction), so an attentive trader gets an early price-action warning to exit before a crash / cover before a spike. It's a pure timing split ŌĆö `preFactor = factor^MEGA_PRE_FRAC` ramps early (via `pushJump`'s optional forced ramp length), the remainder `factor/preFactor` hits at `hits`, so the total factor and `rnd()` order are unchanged. After the peak the move **fades back**: starting at `fadeStart = hits + dur` (end of the boost drift) a slow ~75 s reverting ramp gives back `MEGA_FADE_FRAC` (~two thirds) of the move toward `netTarget = 1 + (factorŌłÆ1)┬Ę(1ŌłÆMEGA_FADE_FRAC)`, so a +30% spike settles around +10% and a panic recovers most of the way ŌĆö the spike is a real selling window, not a new plateau. `fadeTicks` is clamped to the room left before game end (late megas fade faster, or are skipped if there's no room), and again no extra `rnd()` is consumed.
- Local persistence lives in two `localStorage` keys, both fully optional (everything is wrapped in `try/catch`, so a blocked/absent store just disables it): `spcx-duell` holds the last player names + the best-ever P&L record (shown on the start screen, updated in `endRound` excluding tutorial), and `spcx-duell-game` is a resumable snapshot of the running game. The snapshot stores only game *state* (`mode`, `gameCode`, `durationMin`, `round`, `startAt`, `tickCount`, deep `players`, `phase`) ŌĆö never the market, which `genMarket` rebuilds deterministically from the code. `saveSnapshot()` runs each tick/trade/`startRound` (skipped for tutorial and once `over`, except the `"handover"` phase); `clearSnapshot()` fires at match end and on new game. Resume (`resumeBtn`) rebuilds the market, restores state, and continues ŌĆö remote then catches up via the wall-clock anchor; the `"handover"` phase re-shows the pass-over screen via `showHandover()`. Snapshots older than 24 h are ignored.
- Per-round trading stats live in `players[i].stats` (filled in `trade()`/`processTick()`) and feed the awards on the handover/result screens ŌĆö one ordered `AWARDS` list yields the main title (`playerTitle()`: first match) and up to three badges (`playerBadges()`: further non-`tOnly` matches; the duel winner additionally gets `KING_BADGE` prepended) ŌĆö plus `statsLine()` and the detailed `buildAnalysis()` table in the result modal.
- In remote mode the duel comparison works without a server via `packResult()`/`unpackResult()`: a versioned base64 string (currently `SPCX3.ŌĆ”`) carrying result + stats that players exchange manually; `renderCompare()` then shows the same two-column result view as local mode. The payload embeds `gameCode` so results from a different game are rejected, and externally supplied strings (name, favorite stock) must go through `esc()` before touching `innerHTML`.
- The chart is hand-drawn on a `<canvas>` in `drawChart()` (rolling ~3-minute window, devicePixelRatio-aware); its drawing height follows the element's `clientHeight`, so CSS controls it (taller on wide screens). A `resize` listener redraws. `drawChart()` is pure rendering ŌĆö it consumes no `rnd()`, so anything here is fairness-neutral.
- A `requestAnimationFrame` loop (`startChartLoop()` in `startRound()`, `stopChartLoop()` in `endRound()`) redraws the chart at ~60 fps so the curve grows smoothly instead of jumping once per tick. `drawChart()` interpolates the most recent point/candle via `prog = (now ŌłÆ lastTickAt)/TICK_MS` (clamped 0ŌĆō1); `lastTickAt` is set on every `tick()` and `startRound()`. When `paused`/`newsPaused`/`over`, `prog` stays 1 so the last segment shows complete (no animation while frozen). The loop runs in both modes; remote already advances `tickCount` by wall clock, the rAF loop only governs *rendering* smoothness.
- The chart has two views toggled by `chartMode` (`"line"` / `"candle"`, buttons `#chartToggle .ctg`): the line view (area-filled curve) and a candlestick view that aggregates ticks into ~10-second candles (`CT = round(10000/TICK_MS)` ticks each, aligned to absolute tick boundaries so they don't jiggle as the window scrolls). The rightmost (open) candle uses the animated `livePrice` as its close and is drawn semi-transparent; candle color is per-candle (close vs. open). `chartMode` is view-only state, not persisted in the snapshot, and never touches the market.
- Layout is mobile-first (single column, `.wrap` capped at 720px). A `@media (min-width:920px)` block turns the match screen into a two-column grid: the match `.wrap` is split into `.col-chart` (watchlist + quote + chart, left) and `.col-side` (order + depot + news, right), the topbar widens to match, the start-screen form stays narrow (480px), and the canvas grows to 300px. The two column wrappers just stack on phones, so mobile order is unchanged.
- The tutorial (start-screen button `tutBtn`) is a guided practice mode behind the `tutorial` flag: `genTutorialMarket()` returns a hand-scripted, deterministic mini market in the same `{paths, events, tips}` shape as `genMarket` (jumps pre-baked into the paths at display tick + `REACT_TICKS`), and a coach bar (`TUT_STEPS`/`tutShow`) walks through six steps. Explanation/action steps freeze game time via `paused`, which is why the script's fixed event ticks always line up with player progress. Hooks: `tutOnTrade` at the end of `trade()`, `tutOnTick` in `tick()`'s local branch, `tutFinish` in `endRound()` ŌĆö `genMarket` and the duel logic stay untouched.
- **Sandbox** is a free-play sub-mode within Einzelspieler, controlled by the `sandbox` boolean. The start screen shows a `#soloSub` row (­¤Ä» Normal / ­¤Å¢’ĖÅ Sandbox) and a `#capField` (10k/25k/50k/100k $) that only appears in Sandbox. `START_CASH` is a `let` and is set at game-start: `sandboxCash` in sandbox, 25 000 in all other modes.
