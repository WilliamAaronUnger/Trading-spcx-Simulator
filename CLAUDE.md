# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Trading Duell: a two-player, pass-and-play stock trading game around a fictional SpaceX IPO. Both players trade the *exact same* pre-generated market one after the other; whoever ends with the higher P&L wins. It ships as an offline-capable PWA. All UI text, comments, and number formatting are in **German** (`de-DE` locale) — keep new user-facing text and code comments in German to match.

## Development

There is no build system, package manager, or linter. The app is plain HTML/CSS/JS with zero dependencies. Tests exist as standalone Node scripts (no deps, Node ≥ 22): `node worker.test.js` (online API), `node e2e.test.js` (multi-device end-to-end incl. share links), and `node career.test.js` (the endless career-market primitive: determinism, epoch continuity, carry-cache) — run the relevant ones before pushing logic changes; UI changes are still verified by playing. CI (`.github/workflows/test.yml`) runs them, but only via manual dispatch.

- Run locally: serve the directory with any static server (e.g. `python3 -m http.server`) and open `index.html`. Opening the file directly also works, except the service worker only registers over http(s).
- Verify UI/gameplay changes by playing in a browser; logic around the online layer/worker/share links is covered by the two test scripts above.

## Architecture

The application is split across five hand-maintained files (no build step): `index.html` (markup/structure only), `styles.css` (all CSS, formerly the inline `<style>`), `data.js` (the content **data** — `STOCK_DEFS`/`ETF_DEF`, all news/event pools `NEWS_POOL`/`MEGA_POOL`/`MOMENTUM_POOL`/`CHAIN_POOL`/`GENERIC_*`/`OPENING_EVENT`, plus `TIPS`, `AWARDS`, `TUT_STEPS`, `MODE_HINTS`, `DURATIONS`, `DEFAULT_FAVS`, `KING_BADGE` and the tuning constants `TICK_MS`/`TICK_SCALE`/`REACT_TICKS`/`FEE_PCT`/`DIV_PCT`/`DIV_PAYOUT`/`MEGA_*` and the small data accessors `defOf`/`isDividendSym`/`DISPLAY_SYMS`/`feeOf`), `engine.js` (the SHARED deterministic core: `mulberry32`/`genMarket`/ETF derivations, the expert-mode impact math `buildEffPaths`/`findSqueezes`/`spreadAtTick`/`haltLeftAt`, and the anti-cheat `replayRound`/`oracleMaxPnl` — loaded by the browser as a classic script AND by the worker via `import`; both `data.js` and `engine.js` end with a guarded `Object.assign(globalThis, {...})` publish block so the worker import path sees the symbols — keep those lists in sync when adding engine-relevant constants/functions), and `game.js` (all remaining logic: helpers, state, tick loop, trading, rendering, news feed, round/match end, stats, tutorial, controls). A small self-contained helper `qr.js` (dependency-free QR-code encoder + `drawQR`/`qrMatrix`, byte mode, versions 1–10, EC-level M) renders the invite `?join=…` link as a scannable QR in the lobby; it has no game dependencies. `jsqr.js` is the one vendored third-party file (jsQR v1.4.0, Apache-2.0, unmodified below its header): a QR *decoder* that the in-app scanner lazy-loads only where the native `BarcodeDetector` API is missing (e.g. iOS Safari) — don't hand-edit it. `worker.js` is the online ROOM backend (Cloudflare-Worker mini-API v5, imports `data.js`+`engine.js`: persistent rooms with members/presence/roles, creator-started round series each with a fresh secret seed (plus per-round `expert`/`cash` options), one aggregate GET, per-round results feeding an evening scoreboard, and — in expert rounds — a server-stamped anonymous block-order journal (`trades`) with per-player rate limit; plus ANTI-CHEAT: a result PUT must carry the full trade log (`{res, log}`), is only accepted after round end, and the server REPLAYS the log on its own regenerated market (`replayRound`) — the server-computed P&L is what enters scoreboard/ranking, mismatching claims and impossible orders are hard-rejected (422), old clients without a log get 400 (deliberate update requirement), results near the oracle ceiling (`oracleMaxPnl`) get a 🤨 `sus` flag, and logs whose timing only a machine could produce (`botSuspicion` in `engine.js`: entries placed shortly before *unannounced* news, or near-instant reactions across many events — deliberately conservative thresholds so fast honest players aren't flagged) get a 🤖 `bot` flag — tested via `node worker.test.js`); the PWA never loads it (not in `sw.js` FILES). ONLY the `room` mode talks to it (plain `fetch` against `ONLINE_API` in `data.js`); solo/local/remote make ZERO network requests (proven in `e2e.test.js`). Round seeds are decoupled from the room code (`marketSeed`; `SPCX6` payloads carry the seed so mismatched markets are rejected). Details: `RAUM-PLAN.md`/`ONLINE-PLAN.md`. The other files are PWA scaffolding (`sw.js`, `manifest.json`, icons).

These are **classic** scripts (not ES modules), loaded `qr.js`, `data.js`, then `engine.js` **before** `game.js` at the end of `<body>`; both share one global lexical scope, exactly as the former single inline script did — so top-level `const`/`let` in `data.js` are visible to `game.js` (and vice-versa at call time). No `import`/`export`. Keep `file://` opening working by not introducing modules. When adding/renaming any of these files, update `sw.js`'s `FILES` list **and** bump its `CACHE` version. `data.js` must stay free of parse-time references to `game.js` symbols (its internal order already satisfies dependencies, e.g. `TICK_MS` before `MEGA_REACT_TICKS`, `STOCK_DEFS` before `defOf`).

`game.js` is organized into commented sections (`/* ====== ... ====== */`) in this order: helpers, state, tick loop (incl. the expert-mode client wrappers), trading, rendering, news feed, round/match end, controls; market generation lives in `engine.js`.

### Core design: deterministic pre-generated market

The fairness guarantee of the game depends on one idea: the full market (all price paths and all news events) is generated **up front** in `genMarket(seed, ticks)` before round 1 starts, using a seeded PRNG (`mulberry32`). Both rounds then merely *replay* this same data via `tick()` (one tick every `TICK_MS` = 1000 ms; pool data is calibrated for 700 ms ticks and rescaled via `TICK_SCALE`). Consequences when editing:

- Anything random that affects prices or news must use the seeded `rnd()` inside `genMarket`, never `Math.random()`, or the two rounds diverge and the duel becomes unfair.
- `price(sym)` is just an index lookup into `market.paths[sym][tickCount]`; live game code never mutates prices. (Expert rooms are the one sanctioned exception: `price()` reads `effPaths` — a *derived* overlay recomputed deterministically from the server journal; `market.paths` itself is still never touched, and `basePrice()` always returns the fair path.)
- News events carry a `jump` (price shock) and a `drift` over `dur` ticks. The event is *displayed* at its `tick`, but the jump and drift only hit the price `REACT_TICKS` (~8 s) later — that reaction window is the point; don't apply effects at the display tick.
- The 6-digit game code (`gameCode`) *is* the market seed: entering the same code on another device reproduces the identical game. `code % 3` encodes the duration (5/10/15 min via `DURATIONS`), so seed and tick count always match across devices. Don't change `genMarket`'s consumption order of `rnd()` casually — it would silently change what every shared code produces.
- Creating a game opens a lobby overlay; round 1 starts automatically at the *second-next* full wall-clock minute (`openLobby`). Devices that create/join with the same code within the same minute therefore start simultaneously without any server.

### Other things to know

- There are three game modes, held in `mode`. The start screen presents a two-level chooser: a top row (`.mtop`, **Einzelspieler** / **Mehrspieler**) and, when Mehrspieler is picked, a sub row (`.mode`, **Gleiches Gerät** / **Zwei Geräte**); `setTop()` drives the top row and calls `setMode()`. A short `#modeHint` line (`MODE_HINTS`) explains the current pick. The three resulting `mode` values: **`"solo"`** (Einzelspieler) — single player, one round, plays for themselves; **`"local"`** (Mehrspieler · gleiches Gerät) — the original pass-and-play: `players` has both entries, round 1 → handover overlay → round 2 → `showResult()` compares the two with a crown; **`"remote"`** (Mehrspieler · Offline) — one player per device coupled only by the shared code, minute-aligned lobby start, strictly offline; **`"room"`** (Mehrspieler · Online-Raum) — persistent server room (2–20 players + walls), creator starts rounds with secret seeds, live race + evening scoreboard. Two axes cut across these: **timing** — `"solo"` and `"local"` are tick-based and pausable (and their breaking-news popup pauses via `newsPaused`), only `"remote"` is wall-clock-anchored; **result** — `"solo"` and `"remote"` run exactly one round (`round === 0`) and end via `showResultSolo()` (own P&L), but `"solo"` hides the result-exchange box (`cmpBox`) since there's no opponent. `"solo"` starts immediately (no lobby, random code) and counts toward the local record like `"local"`. Init calls `setTop("single")` so the default is Einzelspieler.
- Player state lives in the `players` array (cash, positions with average entry price, result). `round` indexes the active player.
- Positions support shorting: `pos.qty < 0` is a short with `avg` = average short price. There are three order buttons: buy (covers while short), sell (longs only), and a dedicated short button that opens/increases a short (capped by `maxShortQty`: open short exposure ≤ 1× `totalOf`, no leverage); an order never crosses from long to short or vice versa in one step. Generic formulas like `(cur - avg) * qty` already handle the sign — but percent displays and chart up/down logic must flip for `qty < 0`. The result-exchange payload version (currently `SPCX6.…`) must be bumped whenever stats fields change.
- Game-time differs per mode: **local** is tick-based — the pause button and the breaking-news popup (`newsPaused`) halt `tickCount`. **remote** is wall-clock-anchored — `tick()` derives `tickCount` from `Date.now() - roundAnchor` (where `roundAnchor === startAt` from the lobby), there is no pause button, popups don't block (they auto-close), and skipped ticks are caught up after tab sleep. Anything that must stay in sync across devices has to key off this anchored `tickCount`.
- Each stock has a personality via extra `STOCK_DEFS` traits (`newsMult`, `momentum`, `meanRev`, `spikeP`/`spikeMag`), applied inside `genMarket`'s price loop — still fully seeded. The `char` string is shown under the quote.
- Besides plain news there are two-stage chains (`CHAIN_POOL`: rumor → confirm/deny ~1.5–2.5 min later) and insider tips (`market.tips`): vague heads-ups ~50 s before a real generated event. Tips derive from the seed, so both players/devices get identical ones — fairness holds.
- Mega events (`MEGA_POOL`, ~50% of games, at most one, mid-game): huge jumps (+20–40%, the negative Marktpanik capped at ~−15%) marked `mega:true` on the event. They are deliberately unannounced (excluded from insider-tip candidates) but use a longer fuse — `MEGA_REACT_TICKS` (~20 s) instead of `REACT_TICKS` — so shorts can still cover; that fuse is the safety valve, don't shorten it. `newsMult` is intentionally not applied to mega jumps. Jumps are not applied as a single tick: `pushJump()` ramps anything ≥4% over several ticks (`factor^(1/n)` per tick, ~7–14 s for megas) so big moves rise as a curve — the cumulative factor is unchanged, small news still hit in one tick, and no extra `rnd()` is consumed so fairness holds. Megas additionally get a **pre-tremor** (`fireMega`): `MEGA_PRE_FRAC` (~20%) of the move creeps in over the ~12 ticks *before* the main hit (same direction), so an attentive trader gets an early price-action warning to exit before a crash / cover before a spike. It's a pure timing split — `preFactor = factor^MEGA_PRE_FRAC` ramps early (via `pushJump`'s optional forced ramp length), the remainder `factor/preFactor` hits at `hits`, so the total factor and `rnd()` order are unchanged. After the peak the move **fades back**: starting at `fadeStart = hits + dur` (end of the boost drift) a slow ~75 s reverting ramp gives back `MEGA_FADE_FRAC` (~two thirds) of the move toward `netTarget = 1 + (factor−1)·(1−MEGA_FADE_FRAC)`, so a +30% spike settles around +10% and a panic recovers most of the way — the spike is a real selling window, not a new plateau. `fadeTicks` is clamped to the room left before game end (late megas fade faster, or are skipped if there's no room), and again no extra `rnd()` is consumed.
- **Anti-Cheat (room mode)**: every executed fill is appended to `tradeLog` (`[tick, sym, side, qty, block10]`) in `trade()`/`execPending()`, persisted in the snapshot, and submitted with the result; the client's claimed pnl must match the server replay within a small tolerance. Honest-player parity is guaranteed by construction (same `engine.js` on both sides) and pinned by an e2e test that replays a REAL client trade to the cent — if you change any trading rule in `game.js` (fees, spread, dividends, slippage, halts), `replayRound` in `engine.js` MUST change identically, or every result submission will be rejected. The ranking shows ⚠️ for rejected own results, 🤨 for oracle-suspicious ones and 🤖 for bot-suspicious timing (`botSuspicion`); all flags are suspicion markers, never rejections. Offline modes are untouched (no server, no log).\n- Local persistence lives in two `localStorage` keys, both fully optional (everything is wrapped in `try/catch`, so a blocked/absent store just disables it): `trading-duell` holds the last player names + the best-ever P&L record (shown on the start screen, updated in `endRound` excluding tutorial), and `trading-duell-game` is a resumable snapshot of the running game. The snapshot stores only game *state* (`mode`, `gameCode`, `durationMin`, `round`, `startAt`, `tickCount`, deep `players`, `phase`) — never the market, which `genMarket` rebuilds deterministically from the code. `saveSnapshot()` runs each tick/trade/`startRound` (skipped for tutorial and once `over`, except the `"handover"` phase); `clearSnapshot()` fires at match end and on new game. Resume (`resumeBtn`) rebuilds the market, restores state, and continues — remote then catches up via the wall-clock anchor; the `"handover"` phase re-shows the pass-over screen via `showHandover()`. Snapshots older than 24 h are ignored.
- Per-round trading stats live in `players[i].stats` (filled in `trade()`/`processTick()`) and feed the awards on the handover/result screens — one ordered `AWARDS` list yields the main title (`playerTitle()`: first match) and up to three badges (`playerBadges()`: further non-`tOnly` matches; the duel winner additionally gets `KING_BADGE` prepended) — plus `statsLine()` and the detailed `buildAnalysis()` table in the result modal. One badge is expert-room-only: `🎯 Gegen den Strom` accrues `stats.contra` in `noteClose()` when a position is closed at a profit *against* a pronounced room herd skew (`skewNow(sym)` ≥ `SKEW_MIN`, opposite sign to the position) — rewarding contrarian play over pure news-chasing. `contra` is intentionally NOT packed into the `SPCX6` payload (no version bump), so the badge shows for one's own result but not for cross-device opponents; `skewNow()` returns 0 outside the expert room, so the check is inert everywhere else.
- In remote mode the duel comparison works without a server via `packResult()`/`unpackResult()`: a versioned base64 string (currently `SPCX6.…`; `SPCX4`/`SPCX5` stay readable) carrying result + stats that players exchange manually; `renderCompare()` then shows the same two-column result view as local mode. The payload embeds `gameCode` so results from a different game are rejected, and externally supplied strings (name, favorite stock) must go through `esc()` before touching `innerHTML`.
- The chart is hand-drawn on a `<canvas>` in `drawChart()` (rolling ~3-minute window, devicePixelRatio-aware); its drawing height follows the element's `clientHeight`, so CSS controls it (taller on wide screens). A `resize` listener redraws. `drawChart()` is pure rendering — it consumes no `rnd()`, so anything here is fairness-neutral. `drawChart(o)` takes an optional opts object so callers other than the live game can reuse the exact same renderer: `o.canvas`/`o.sym`/`o.market`/`o.tick`/`o.pos`/`o.chartMode` override the globals, `o.prog` supplies an external (wall-clock) sub-tick interpolation instead of the `lastTickAt` one, and `o.big` scales fonts/markers (factor `S`) up for the beamer/TV. The Leinwand (`wallFrame`) drives it this way at `big:true` with `pos:null`.
- A `requestAnimationFrame` loop (`startChartLoop()` in `startRound()`, `stopChartLoop()` in `endRound()`) redraws the chart at ~60 fps so the curve grows smoothly instead of jumping once per tick. `drawChart()` interpolates the most recent point/candle via `prog = (now − lastTickAt)/TICK_MS` (clamped 0–1); `lastTickAt` is set on every `tick()` and `startRound()`. When `paused`/`newsPaused`/`over`, `prog` stays 1 so the last segment shows complete (no animation while frozen). The loop runs in both modes; remote already advances `tickCount` by wall clock, the rAF loop only governs *rendering* smoothness.
- The chart has two views toggled by `chartMode` (`"line"` / `"candle"`, buttons `#chartToggle .ctg`): the line view (area-filled curve) and a candlestick view that aggregates ticks into ~10-second candles (`CT = round(10000/TICK_MS)` ticks each, aligned to absolute tick boundaries so they don't jiggle as the window scrolls). The rightmost (open) candle uses the animated `livePrice` as its close and is drawn semi-transparent; candle color is per-candle (close vs. open). `chartMode` is view-only state, not persisted in the snapshot, and never touches the market.
- Layout is mobile-first (single column, `.wrap` capped at 720px). A `@media (min-width:920px)` block turns the match screen into a two-column grid: the match `.wrap` is split into `.col-chart` (watchlist + quote + chart, left) and `.col-side` (order + depot + news, right), the topbar widens to match, the start-screen form stays narrow (480px), and the canvas grows to 300px. The two column wrappers just stack on phones, so mobile order is unchanged.
- The tutorial (start-screen button `tutBtn`) is a guided practice mode behind the `tutorial` flag: `genTutorialMarket()` returns a hand-scripted, deterministic mini market in the same `{paths, events, tips}` shape as `genMarket` (jumps pre-baked into the paths at display tick + `REACT_TICKS`), and a coach bar (`TUT_STEPS`/`tutShow`) walks through six steps. Explanation/action steps freeze game time via `paused`, which is why the script's fixed event ticks always line up with player progress. Hooks: `tutOnTrade` at the end of `trade()`, `tutOnTick` in `tick()`'s local branch, `tutFinish` in `endRound()` — `genMarket` and the duel logic stay untouched.
- **Sandbox** is a free-play sub-mode within Einzelspieler, controlled by the `sandbox` boolean. The start screen shows a `#soloSub` row (🎯 Normal / 🏖️ Sandbox) and a `#capField` (10k/25k/50k/100k $) that only appears in Sandbox. `START_CASH` is a `let` and is set at game-start: `sandboxCash` in sandbox, 25 000 in all other modes. The market is generated for 60 minutes so it never runs out. Auto-end is suppressed in `tick()` (`!sandbox` guard); the player ends manually via `#endSandboxBtn` (⏹ Beenden). Like `tutorial`, sandbox skips `updateRecord()` and `saveSnapshot()`. The countdown displays elapsed time (counting up) instead of time remaining. `applySoloUI()` manages all visibility toggles for the solo sub-row; it is called from `setMode()` so it stays in sync whenever the mode changes.
- **🎓 Experten-Modus** (`expert` flag, full design in `IMPACT-PLAN.md`): opt-in per room round (creator-only toggle + `cash` preset in the round resource, so every device plays identical rules) and per sandbox game (`sandboxExpert`). Two layers: **(a) room-only dynamic market** — block orders (≥ `BLOCK_MIN_FRAC` of `START_CASH`) are POSTed to the worker, server-stamps them (`at`), and every client (incl. the Leinwand) translates the anonymous journal into `effPaths` via `buildEffPaths()`: impact ramps in `REACT_TICKS` after the stamp (announcement before effect, same grammar as news), fades ~2/3 back, capped ±`IMPACT_CAP`, per-stock depth via the `liq` trait; herd skew (`skewAt`) dampens moves in the crowd's profit direction and news against a ≥`SKEW_MIN` skew spawn a synthetic squeeze impact (`findSqueezes`). Own fills pay half their own impact (slippage → `stats.slip`), and `endRound` settles at `basePrice` („Schlussauktion") so last-second pumps are worthless. The index values `ETF_SYM`/`ETF2_SYM` (MKT/ACT) get NO own overlay — `buildEffPaths` skips them in the impact loop and re-derives them from the *effective* constituent paths (`addEtfPath`/`addActivePath` over `eff`), so the index inherits its stocks' impact but can't itself be pushed by trading it (they carry no own news/squeeze either, since events never target them). Index trades are consequently exempt from block treatment entirely (`isIndexSym` guards the `blockVol` paths in `trade`/`execPending` **and** the replay's `unblocked` check, in lockstep): no impact slippage, no journal entry, no 🐘 announce — you still pay the normal spread. All of this consumes zero `rnd()` — fairness by identical journal. **(b) local hardships** (also sandbox): bid/ask spread (`spreadOf`, widened ×3 for ~30 s after news), trading halts after mega panics (`haltInfo`, ~15 s), limit/stop orders (`placeOrder`/`execPending` in `processTick`, max 4; the order panel is collapsed by default behind `#ordToggle` — progressive disclosure via `ordPanelOpen`/`applyOrdPanel()`, with an open-order count on the toggle — so the base buy/sell/short surface stays calm), shorts pay the dividend (negative `pendingDiv`), ACT holding costs. The tutorial never runs expert; the local record ignores expert rounds; expert/cash live in the snapshot; `SPCX6` payloads carry expert flag, cash, slippage and a journal hash (ranking flags mismatches).
- **Karriere-Modus** (`mode === "career"`, full design in `KARRIERE-PLAN.md`): an isolated extra mode — a persistent, real-time **wealth tycoon** reached via its own `#careerBtn` on the start screen (not the normal match form). You build an empire in wall-clock time (also offline): a monthly basic income (`CAREER_BASIC_INCOME`) plus income from purchased assets (`CAREER_ITEMS`, categorized via `CAREER_CATS` into Immobilien/Unternehmen/Finanzen/Luxus, each with an `income`/month and escalating per-unit cost `CAREER_COST_MULT`, buyable in unlimited quantities) accrue into cash every `CAREER_MONTH_MS` (~5 min = 1 "month", `careerAccrue`/`careerMonthlyIncome`); the shop is the main loop and **trading is a demoted side show** (`#careerTradeBtn` → the market, income keeps flowing into cash while you trade). Bust (`net < CAREER_MIN`) → bailout to `CAREER_START`, assets kept. The hub ticks income live (`careerHubTimer`), the market runs by wall clock even while the app is closed. Endless via deterministic epochs with price-carry: `epochSeed`/`careerCarry`/`careerMarket` in `engine.js` (pure, proven by `career.test.js`) chain `genMarket` segments so epoch e starts where e−1 ended (MKT/ACT re-derived from carried constituents). Own persistence key `trading-duell-career` (`{seed, anchor, cash, pos, owned, peakNet, carry, carryEpoch, …}`); the market is recomputed from seed+anchor+now, positions/cash persist. `wallClock()` includes career (no pause, non-blocking popups); the career branch in `tick()` derives the epoch offset from the clock, rebuilds the market on epoch change (`careerSyncToNow`), advances small gaps smoothly and jumps big ones (no downtime dividends/event-replay). Winnings buy cosmetic-only `CAREER_ITEMS` (one icon each, else plain text — emojis deliberately sparse) shown in a cabinet; text-only rank titles from `CAREER_RANKS` by peak net worth. Fully isolated: excluded from `updateRecord`/`appendGameHistory`/`saveSnapshot`, never calls `endRound` (leaves via `leaveCareerToHub` → hub), touches neither the record nor the competitive modes.
- `sw.js` uses a network-first/cache-fallback strategy with a versioned cache name (`trading-duell-v<N>`). When changing cached assets in a way that must reach installed PWAs, bump the `CACHE` version string, and keep the `FILES` list in sync with any added/renamed assets.
